== 檔案與目錄
軟體在媒體上是以樹狀的邏輯形式儲存，作業系統則將其視為檔案與目錄的結構來管理。
關於 Unix 類的作業系統，有一句常常會提到的說法：「一切都是檔案」。
雖然並非百分之百適用，但是做為揭示 Linux 作業系統架構的特色，仍然是十分恰當的描述。

=== 檔案系統
檔案系統是對於磁碟分割區的格式化劃分。
如果將分割區看成是所有書頁未切割前拼成的一張大紙，那麼檔案系統就是書的編排格式。
譬如每頁的大小、每行或每列的間隔與長度，都會隨格式不同而有差異。
就像是漢字書籍可以用直式或橫式排版一樣，分割區上的檔案系統也十分多樣，卻都能以不同的形式提供資料的存取。

==== 種類
基於開發歷史、存取效率、功能需求、專利限制…等原因，每個作業系統都有其主流的檔案系統。
至於是否能夠支援舊式、罕用或是其他作業系統的檔案系統，取決於能否載入適當的驅動程式。
網路儲存裝置也有它們專用的檔案系統，通常是由負責的網路協定所決定。

==== 路徑
Linux 是以**路徑**(path) 來指明檔案或目錄在檔案系統中的位置。
路徑是由一連串的目錄與檔案名稱所組成，彼此以斜線分隔，並依照檔案系統中的層級關係排列。

完整的路徑是由斜線起始，表示檔案系統的**根目錄**(root directory)。
然後從檔案系統樹狀結構的根部開始，列出途中經過的目錄名稱，最後到達目標的檔案或目錄。
這種完整的路徑表示方法稱為**絕對路徑**，而不以斜線起始的表示方法則稱為**相對路徑**，意思是指相對於當前工作目錄的位置。
路徑的最大長度為 4096 字元，而不包含任何斜線的單一名稱，可以算是最簡化的相對路徑。如果沒有特別強調，提到路徑時就包括了絕對路
徑與相對路徑，當然也包括了單一檔案名稱與單一目錄名稱。

=== 檔案系統階層
作業系統於執行期間所能控制的檔案系統，是一個整體的**虛擬檔案系統**(VFS)，其樹狀結構即稱之為**檔案系統階層**(file system hierarchy)。
就像一棵由根目錄起始的大樹，向上分支出如同枝幹的目錄，而目錄中還會長出如同樹葉的檔案，或是再分支出子目錄。
由於檔案系統階層是一個抽象的概念，因此有可能涵蓋了多個儲存分割區上的檔案系統，同時包括了進入系統作業階段才建立的假擬目錄。

image::file-sys-hier.svg[width=70%,align="center"]

在 Linux 檔案系統階層中，根目錄之後是一些屬於第一層的子目錄：
====
user$ *ls -F /* +
bin@　 dev/　home/　lib64@　mnt/　proc/　run/　 srv/　tmp/　var/ +
boot/　etc/　lib@　 media/　opt/　root/　sbin@　sys/　usr/
====
其中附加了斜線(`/`) 的名稱是標準的目錄，而附加了**在號**(`@`) 的名稱表示衹是一個符號鏈結。
即使不利用 `-F` 選項來為目錄與鏈結名稱附上辨識符號，目前系統預設的 `ls` 命令別名也會以顏色來區分不同類型的檔案或目錄名稱。
如果想看看檔案系統階層的模擬樹狀結構，外部命令 `tree` 就可以滿足這個需求。
以下粗略地將這些目錄分為三類來介紹：系統目錄、個人目錄、假擬目錄。

==== 系統目錄
系統開機時需要存取的**內核**(kernel) 與**初始記憶體檔案系統**(initramfs) 或逐漸被棄用的**初始記憶體磁碟**(initrd)，是系統運行時期最重要的作業核心，一般都會放在 `/boot/` 目錄之中。
內核的映像壓縮檔案名稱通常是 `vmlinuz` 再加上版本號碼，以便與有可能共存的其他版本做區分。
內核初始時所需要的根檔案系統，是由初始記憶體檔案系統所提供，目錄中還包括其他同樣依內核版本區分的附帶檔案。
由於內核所在的檔案系統基於安全的考量而限制較多，通常會將 `boot/` 從 `/` 之下分離出來，做為獨立分割區的掛載點。

目前主流的 UEFI 開機模式，它的開機分割區需要 `vfat` 格式的檔案系統，
因此 Linux 的檔案系統階層，會將開機分割區（假設是 `/dev/sda1` 裝置）掛載於 `/boot/efi/` 目錄之下，
而主要資料所在的根分割區（假設是 `/dev/sda2` 裝置）則掛載於 `/` ，成為檔案系統階層的根目錄。

傳統的 Unix 系統中，開機過程有可能用到的命令程式會放在 `/sbin/` 或 `/bin/` 路徑內，而前者所含的程式通常需要**超級使用者**(root) 的身分才允許執行。
適合較新 64 位元硬體架構的軟體函式庫，會存放在 `/lib64/` 路徑之下，而適合舊式 32 位元硬體架構或與硬體架構無關者，則會置於 `/lib/` 路徑之下。

`sbin`, `bin`, `lib`, `lib64` 等目錄名稱在 `/usr/` 路徑下會再次出現，這是由於傳統大型主機將 `/usr/` 做為獨立分割區的掛載點，甚至會使用網路檔案系統來掛載。
一些開機時期非必要、一般使用者才會用到的程式與函式庫，就被放到了 `/usr/` 之下相對應的子目錄中。
但是經過了在個人電腦上的演變影響，分離掛載的 `/usr/` 路徑已無太大的必要性，甚至還出現了將 `sbin/` 合併入 `bin/` 目錄的提議。
因此目前較流行的做法，是維持 `/usr/` 與 `/` 在同一個分割區中，並且避免直接引用根目錄之下的前述四個目錄。
亦即將它們做為 `/usr/` 路徑下各自同名目錄的符號鏈結（參閱本章稍後的〈檔案屬性〉一節），以維持與傳統軟體的相容性。

由此可知 `/usr/` 是檔案系統階層中最主要的分支，而除了命令程式與函式庫之外，
還包括像是 `/usr/include/` 之下的軟體開發 C 語言**標頭檔案**(header files)，以及存放了各類資料檔案的 `/usr/share/` 路徑。
另外 `/usr/libexec/` 則儲存了一些雖然是可執行檔案，但是實際上不適合做為命令直接呼叫，而是由其他命令間接執行的程式。

系統中各項軟體的組態預設值，一般會儲存在 `/etc/` 路徑的同名檔案或子目錄之中，做為該軟體的**全域設定值**(global settings)。
軟體在執行期間所動態產生的資料，通常會儲存在 `/var/` 路徑下依照軟體性質與名稱分類的子目錄之中。
又如傳統上儲存於 `/srv/` 之下的各類伺服器資料，許多已經轉移到 `/var/` 或 `/var/lib/` 中使用。
而某些特殊用途的軟體，仍然會將自身的整套系統，儲存於 `/opt/` 路徑的子目錄之中。

==== 個人目錄
每次的命令殼作業階段，都會隨時以 `PWD` 變數，來記錄一個稱為**工作目錄**的值，做為使用者目前在檔案系統上所處位置的指標。

登入後的預設值是**家目錄**(home directory)，而一般使用者的家目錄就是在 `/home/` 路徑之下，與使用者賬號同名的子目錄。例如使用者 user 的家目錄就在 `/home/user/` ，但是超級使用者的家目錄特別預設為 `/root/` 。
家目錄的位置會依照登入身分記錄在 `HOME` 變數中，而指令中的路徑如果是以波浪號(`~`)開頭，且做為單一目錄名稱使用的話，則會被命令殼代換為家目錄。
舉例來說， `~/Downloads/` 代表的路徑等同於 `$HOME/Downloads/` ，都是表示標準的個人下載目錄。

家目錄中還有一些子目錄對映著系統目錄的角色，但是僅限於該使用者讀寫，而且衹對該使用者有作用。
例如 `$HOME/bin/` 中含有可執行檔案， `$HOME/.local/share/` 中含有各類資料檔案。
許多個人化的軟體組態檔位於 `$HOME/.config/` 之中，做為**本地組態檔**(local configuration) 而類似 `/etc/` 之於整個系統的儲存位置。

==== 假擬目錄
由於 Linux 是多人多工的作業系統，因此每位使用者都受到所屬身分的限制，尤其反映在對於檔案的存取能力上。
一般使用者除了家目錄以下具有完整的檔案讀寫權利之外，系統中大部分檔案對他而言都是唯讀的。
但是 `/tmp/` 目錄較為特殊，一般使用者均允許在其中建立衹有自己才能讀寫的目錄與檔案。
做為暫時性的檔案存放位置，路徑 `/tmp/` 目前預設衹存在於記憶體檔案系統中，包含的資料於關機後就會消失。

Linux 系統上的硬體，都是以裝置節點的形式，出現在檔案系統階層的 `/dev/` 路徑之下。
例如 `/dev/sda1` 代表了第一顆 SCSI/SATA/USB 磁碟上的第一個分割區，而 `/dev/mmcblk1p2` 代表了第二顆 eMMC/SD 磁碟上的第二個分割區。
每種裝置都有各自獨特的命名規則與初始編號數字，而這些儲存裝置分割區將會被**掛載**(mount) 到主要的 `/`, `/boot/efi/` 或是次要的 `/mnt/`, `/media/` 路徑之上。

**行程**(process) 表示系統運行時的個別執行程序，而作業系統運行階段的所有行程，集合起來稱之為行程階層。
行程階層也是抽象性的樹狀結構，完全衹存在於系統作業階段。
檔案系統階層中的 `/proc/` 路徑以下，是行程階層的一種映射型式，動態對應著個別行程的資訊。
類似的假擬目錄還包括 `/sys/` ，代表著系統軟硬體階層的映射，同時做為存取各個組成設定值的介面。

命令 `ps` 能夠列出目前由使用者自己所執行的行程，如果加上 `-aux` 選項的話，就可以看到系統中所有行程的詳細列表。行程列表中有一欄標題是 PID，代表該項行程的
流水識別號，常用來做為其他命令的參數使用。
例如：執行 `kill -9 _PID_` 表示向相應的行程發送 KILL(砍除) 訊號，能夠強制結束負責該項行程的程式。
其他常用的類似命令有 `pkill` 與 `xkill` ，分別依程式名稱與滑鼠點選來決定要強制結束的行程。
若是想看到樹狀的行程階層，則需要執行 `pstree` 外部命令，就會發現整個系統的運作都是由 `systemd` 這個行程所帶動的
（這個 PID 為 1 號的初始行程，可能仍然維持著傳統的 `init` 程式名稱）。


=== 文字檔
文字檔是使用電腦時最容易處理的檔案格式，不僅易於閱讀，也方便進行新增、修改、刪除等編輯作業。
文字檔所儲存的內容主要是各類文件，當然也包括了眾多的電腦程式語言原始碼。
即使是圖片或音樂，也有可能以文字描述的格式記錄下來。

==== 一般檔案類型
多數的檔案都支援**魔術字元**(magic number) 的自我標示，也就是以檔案開頭的固定位置、固定個數字元來標示類型。
程式軟體也是根據這個方法，識別收到的資料是否屬於可以處理的對象，而不會依靠為了讓使用者易讀的檔案名稱。
文字模式的 `file` 工具程式，除了利用這個特性之外，還會配合解析檔案內容，讓使用者能夠初步識別參數檔案的類型。
====
user$ *echo _測試資料_ | gzip -c > _檔案.ext_* +
user$ *file _檔案.ext_* +
檔案.ext: gzip compressed data, from Unix, truncated +
user$ *file _/usr/bin/file_* +
/usr/bin/file: ELF 64-bit LSB pie executable, x86-64, ...
====

==== 文字檔類型
如果依照廣義的說法，所謂的文字檔就是：檔案內容需要符合某種字元集的編碼規範，而解碼後所得到的，都是畫面可顯示、使用者可讀的文字。
其中可能包含一些換列、換頁、檔案結尾等簡單的控制字元，但不會出現複雜的非文字格式字元。
狹義的文字檔常被稱為**純文字檔**(text file)，它的編碼符合 *UTF-8*(八位元萬國碼編碼形式) 或是相容的子集合 *US-ASCII*(美國標準資訊交換碼)。

US-ASCII 是固定長度的編碼，衹需要 7 個位元的空間，但是因為一般資訊處理都是以一個位元組（8 個位元）為單位，所以實際的編碼就用一個位元組來儲存一個字元。
UTF-8 是基於**萬國碼**(Unicode) 字元集的一種變動長度的編碼，每個字元會由 1~4 個不等的位元組來表示，而每個漢字通常需要 3~4 個位元組的儲存空間。
外部命令 `iconv` 雖然可以用來轉換編碼，不過是否成功仍取決於原始與目標字元集的相容程度，以及文字內容涵蓋的字元集範圍。

有些軟體為了相容的原因，還在純文字檔的最前方加上兩個字元，用來區別硬體架構的**尾序**(endianness)。
但是這種稱為 *BOM*(位元組順序標記) 的魔術字元，在 Linux 系統上的使用並不普遍。

WARNING: 使用前置 BOM 的純文字檔，可能會造成讀取或執行時的錯誤。

即使是純文字檔這樣相對比較簡單的類型，在不同的作業系統上仍然有細微的差異。
就以換列來說，Linux 或其他 Unix 相容系統是使用 LF(ASCII 10) 字元；舊版 macOS 使用 CR(ASCII 13) 字元，但新版已與 Linux 沒有區別；
而 DOS 與 Windows 系統則使用 CR+LF 兩個字元，做為標示每列結束的符號。
對於 BOM 以及換列的問題，`dos2unix` 及其相關的外部命令，共同提供了方便的格式轉換工具。

==== 處理文字檔
Linux 中許多類型的檔案，都是以文字的格式呈現，或是存在著文字的轉換格式，因此處理文字檔就成了操作系統必備的技能。

最簡單建立文字檔的方法，是透過將命令的輸出重導至一個檔案名稱所產生的。
但是要注意隨著重導符號的不同，會造成內容被覆蓋或是附加的結果。
使用外部命令 `cat`(concatenate) ，可以顯示檔案參數中的內容（也就是輸出到終端機畫面中）。
萬一檔案內容太多，導致前面的資料來不及看清楚就捲出畫面之外，就可以使用 `more` 或 `less` 做為直接或管線命令，讓輸出成為分頁的模式。
====
user$ *echo _初始的內容_ > _檔案.txt_* +
user$ *echo _覆蓋的內容_ > _檔案.txt_* +
user$ *cat _檔案.txt_* +
覆蓋的內容 +
user$ *echo _新增的內容_ >> _檔案.txt_* +
user$ *cat _檔案.txt_* +
覆蓋的內容 +
新增的內容
====

習慣上，**文字編輯軟體**的功能僅限於處理純文字格式，而**文書處理軟體**則用來產生比較複雜的格式文件。對於終端機環境中的文字編輯器，Linux 使用者一直以來都是各有所
好。例如比較常用的 `nano`, `joe`, `vi`, `emacs` 等程式，操作概念與方法都有
不小的差異。使用者需要實際嘗試看看，才能找出適合自己的工具，同時至少了解其他編輯器的基本操作，以備不時之需。
環境變數 `EDITOR` 會指向系統預設的文字編輯器，有些軟體會根據這個變數值來呼叫它們的輔助程式。
使用者可以在 `$HOME/.bashrc` 檔案中，重新將它指定成慣用的命令路徑。

將檔名做為參數來打開有相容問題的文字檔時，每種文字編輯器的表現也可能不盡相同。
以上節所提到含有 BOM 與 DOS 換列格式的檔案為例，就有可能看到以下的編輯畫面：

應該足以提醒使用者要注意，並採取適當的處理方式，再進行任何形式的編輯：
====
user$ *file _檔案.txt_* +
檔案.txt: Unicode text, UTF-8 (with BOM) text, with CRLF line terminators +
user$ *dos2unix _檔案.txt_* +

在圖形環境中，可以選擇的文字編輯軟體就更多了，而文書處理軟體也都有匯入與匯出文字檔的功能。
值得一提的是，不論使用滑鼠雙擊或拖曳將一段文字反白之後，在 Linux 環境中直接按下滑鼠滾輪鍵（中鍵），
就會將文字貼上游標所在的位置，還可以多次操作，這通常要比軟體中原本的拷貝╱貼上功能更為快速與方便。

當要對多個大型的文字檔案進行類似處理，而每個檔案又有多處需要更動的部分時，
以編輯器逐檔載入後修改再儲存，顯然不是一個有效率的操作方式。
以下列舉幾個常用的外部命令，適合批次處理文字檔案：

[square]
* `wc`(word count): 計算檔案內容的列、字詞、字元數量。
* `cut`: 將指定欄位的內容分割出來。
* `sort`: 依照指定順序條件重新排列各列位置。
* `grep`(global regular expression search and print): 篩選出含有指定**胚騰**(pattern) 的文字列。
* `sed`(stream editor): 依照指定胚騰進行內容的新增、刪除、替代操作。
* `awk`(Aho, Weinberger, Kernighan): 以簡單程式語言逐列處理內容。

上述工具大致按照功能由簡單到複雜排列，建議讀者可依需求循序學習。
其中提到的胚騰遵循著**正規表示式**(regular expression) 的規範，是學習進階文字處理必備的知識。
大部分使用者衹有認識基本的萬用匹配，但是它與正規表示式都應該被深入了解，而且重要的是兩者不要搞混。
如果需要混合使用多個工具，實務上會利用管線符號(`|`) 將命令組合起來，甚至會寫成一個簡單的命令稿。
（在試驗階段可能還是需要將個別命令分開執行，才能確認中間產生的暫時檔案內容無誤。）

==== 程式語言檔
可執行檔案是多條指令組成的集合，通常是經由編譯、組譯後得到的二進位格式機器碼，可以直接交給處理器執行；
也可能是由程式語言指令組成的文字格式，由語言的**解譯器**(interpreter) 負責讀入執行；
或是由**位元碼**(bytecode) 組成的中介代碼格式，透過**虛擬機**(process virtual machine) 的載入後間接執行；
甚至是在基於瀏覽器的虛擬平臺上運作。

===== Python 語言
下面是一個以 Python 指令稿語言編寫的九乘九表格輸出程式，先利用文字編輯器將它輸入到檔案：

.9x9.py
....
#!/usr/bin/python3
for i in range(1, 10):
    for j in range(1, 10):
        print(i*j, end=' ')
    print()
....

儲存後就能藉由輸入 `python3 9x9.py` 指令，呼叫 Python 解譯器去執行這個指令稿。
也可以先用 `chmod +x 9x9.py` 指令將其屬性設定為可執行，再輸入 `./9x9.py` 來執行它。
實際過程中，內核仍然會先載入檔案開頭**井歎符號**(shebang)(`#!`) 後方指定的命令檔案，再以指令稿檔案做為它的參數去執行。

===== C 語言
做為對照的 C 程式語言，先將以下內容存入檔案：

.9x9.c
....
#include <stdio.h>
void main() {
    int i, j;
    for(i = 1; i <= 9; i++) {
        for(j = 1; j <= 9; j++) {
                printf("%d ", i*j);
        }
        printf("\n");
    }
}
....

然後呼叫 C 的編譯程式 `gcc` （如果系統中不存在此外部命令，則需要先安裝好相關套件），將它編譯為二進位格式的機器碼檔案。
如果順利的話，就會產生已經帶有可執行屬性的 `9x9` 檔案，可以直接輸入 `./9x9` 來執行它：
====
user$ *gcc _9x9.c_ -o _9x9_* +
user$ *_./9x9_* +
1 2 3 4 5 6 7 8 9 +
2 4 6 8 10 12 14 16 18 +
3 6 9 12 15 18 21 24 27 +
4 8 12 16 20 24 28 32 36 +
5 10 15 20 25 30 35 40 45 +
6 12 18 24 30 36 42 48 54 +
7 14 21 28 35 42 49 56 63 +
8 16 24 32 40 48 56 64 72 +
9 18 27 36 45 54 63 72 81
====

NOTE: 編譯出來的二進位檔案，實務上在安裝前大多會經過 `strip` 命令的處理，藉由刪去除錯用的符號資料，達到縮減檔案的效果。

程式語言衹是一種規範，有多種可能的方式讓它實際運作起來。
常見的 C 編譯程式並非衹有一種，還可以使用 `clang` 做為替代方案。
這裡的 `gcc` 或 `clang` 衹是**編譯器**(compiler) 集合軟體的前端代表程式之一，其與**組譯器**(assembler)、**鏈結器**(linker)…等元件，共同組成一套複雜的**工具鏈**(toolchain)。

至於某種程式語言所寫成的檔案，是否需要經過編譯再執行，或是解譯時即可執行，其實並沒有絕對的規定。
如果能下載到並安裝好 C 語言的解譯程式，就可以不經編譯而直接執行 C 語言程式，當然也可以找找能夠編譯前一小節 Python 程式的軟體來體驗看看差異。


=== 檔案屬性
每個檔案除了本身的名稱以及在檔案系統階層中所處的位置之外，還具備了一些**屬性**(attribute)。
利用後面〈常用命令：目錄操作〉一節中提到的 `ls` 命令，就能夠列出檔案或目錄的屬性：
====
user$ *ls -la _$HOME_* +
總用量 24 +
drwx------ 3 user   user　 4096  4月 16 12:58 . +
drwxr-xr-x 4 root   root　 4096  4月 16 12:58 .. +
-rw-r\--r-- 1 user   user　　 18  9月 27　2025 .bash_logout +
-rw-r\--r-- 1 user   user　　141  9月 27　2025 .bash_profile +
-rw-r\--r-- 1 user   user　　492  9月 27　2025 .bashrc +
drwxr-xr-x 4 user   user　 4096  7月 23　2025 .mozilla
====
列出的資料如果以空格區隔，可以分為 9 欄，依次為：屬性字串、鏈結數量、擁有使用者、擁有群組、資料大小、修改月分、修改日號、修改年分、名稱。

==== 屬性字串
儲存媒體上的一般資料會被視為是正規的檔案，在檔案屬性字串的第一個字元顯示為 `-`(短橫線)。
正規檔案是由一群向後鏈結的區段所組成，彼此在實際的儲存媒體上不必然會緊接在一起。
目錄也可以視為是一種檔案，就像書中的目次也會佔據頁數，其中記載著成員（檔案或目錄）的名稱、屬性與鏈結，它的屬性字串首字元則標記為字母 `d` 。
`/dev/` 目錄下常見的磁碟類型儲存裝置，會被歸類為**區塊**(block) 裝置，於檔案屬性字串的第一個字元標記為 `b` 。
至於大部分的輸入輸出字元類型裝置，則是以字母 `c`(character) 做為檔案屬性字串首字元的標記。

屬性字串除去首字元之後的 9 個字元，決定了檔案或目錄的權限屬性。
其中每 3 個字元為一組，每個字元由左到右分別代表讀取(r)、寫入(w)、執行(x)的屬性，而 3 組則依次規定了這個檔案或目錄的擁有者(u)、擁有群組(g)以及其他人(o)的權限。
權限屬性中如果有字母標記的部分，表示該分組具備該屬性的權利，否則會以 `-` 來表示受到限制。
這三種屬性的權限對於檔案而言，很容易依照字面上的意思來理解，就是限定了是否能夠讀取(r)、寫入(w)以及執行(x)該檔案。
而對於目錄來說，則要注意可以執行(x)是表示允許列出其中所含的檔案或子目錄。
另外在較新的系統中，屬性字串還可能出現第 11 個字元，用來表示與 SELinux 有關的額外存取控制性質。

衹有檔案或目錄的擁有者或是超級使用者，被允許利用 `chmod`(change mode) 命令來修改屬性字串。
`chmod` 的第一個參數是讀寫屬性的設定值，可以接受權限代號或者八進位數字模式之一，而第二個之後的參數則為想要改變的檔案或目錄名稱。
最常使用的 `chmod` 命令選項是 `-R`(recursive) ，表示遞迴進入參數中的目錄以設定屬性。
====
root# *ll _檔案.ext_* +
-rwxrwxrwx 1 user   user   3628  5月 14  2025 檔案.ext +
root# *chmod 644 _檔案.ext_* +
root# *ll _檔案.ext_* +
-rw-r\--r-- 1 user   user   3628  5月 14  2025 檔案.ext +
root# *chmod +x _檔案.ext_* +
root# *ll _檔案.ext_* +
-rwxr-xr-x 1 user   user   3628  5月 14  2025 檔案.ext
====
NOTE: Linux 還支援另一套獨立、功能強大但較少使用的屬性系統，相關的命令包括 `lsattr` 與 `chattr` 。

==== 鏈結與大小
**符號鏈結**(symbolic/soft link) 類似其他作業系統中的捷徑，是僅有一條內容的檔案，註記著建立鏈結的目標檔案或目錄位置。
符號鏈結的檔案屬性字串首字元，是以字母 `l` 做為標記。
設定符號鏈結是利用 `ln`(link) 命令加上 `-s` 選項，以及相對於目標的來源路徑與目標路徑，而取得鏈結的相關命令則為 `realpath` 與 `readlink` 。
較新的 Linux 系統，為了解決安裝某個軟體的多重版本後，如何選擇當前作用版本的問題，而採取的**替代方案**(alternatives) 機制，也是基於符號鏈結的特性。

屬性列表的第二欄，表示與其他檔案或目錄產生的鏈結數量。
對於目錄來說，相當於其中所含子目錄的數量（已經加上 `.` 與 `..` 兩者）。
至於檔案的鏈結數量通常為 1，並不會計入符號鏈結的數量，除非曾經產生過**硬式鏈結**(hard link)。
依照檔案系統的內部處理邏輯，鏈結數量降為 0 時，就表示檔案或目錄已經完整刪除。

當任何命令輸出資料的大小資訊時，通常是顯示實際內容所使用的位元組數量。
由於資料在媒體上是以區塊的形式儲存，因此檔案或目錄實際佔用的空間，要略大或等於它們實際的內容大小。
另外 `/dev/` 假擬目錄下的裝置節點有一項特徵，就是其資料大小欄位會被主裝置號與次裝置號所替代。

==== 擁有權
簡單來說，檔案或目錄的建立者就是它的擁有者，而該使用者所屬的群組，就成為了此檔案或目錄的預設擁有群組。
如果使用者的身分，並不符合屬性列表第三、第四欄所列出的擁有者或擁有群組，那麼對於這個檔案或目錄而言，就衹能做為其他人來看待。

變更檔案或目錄的擁有權需要透過 `chown`(change ownership) 命令，同樣衹允許擁有者或超級使用者才能執行。
第一個參數是擁有權的設定值，由新的擁有者與擁有群組以冒號相連而成，而第二個之後的參數則為想要改變的檔案或目錄名稱。
`chown` 最常使用的命令選項是 `-R`(recursive) ，表示遞迴進入參數中的目錄以設定擁有權。
====
root# *ll _檔案.ext_* +
-rwxr-xr-x 1 user   user   3628  5月 14  2025 檔案.ext +
root# *chown root:root _檔案.ext_* +
root# *ll _檔案.ext_* +
-rwxr-xr-x 1 root   root   3628  5月 14  2025 檔案.ext
====

==== 修改日期
屬性資料第 6, 7, 8 欄的作用，是顯示檔案或目錄最近一次的修改時間(change time)。
如果加上額外的選項，還可以選擇改為顯示它的建立時間(birth time)，或是最近一次的存取時間(access time)。
顯示的前兩欄分別代表月分與日號，第三欄通常是顯示年分，但是如果與現在時間相差半年以內的話，則改為顯示時與分的值。

外部命令 `touch` 加上檔案或目錄名稱做為參數，以及透過 `--date=_日期時間_` 選項設定的字串，可以用來修改時間的屬性資料。
假如工作目錄中不存在 `_檔案.ext_` 檔案，那麼在工作目錄中執行 `touch _檔案.ext_` 指令，就會建立一個名為 `_檔案.ext_` 的空檔案，
與 `echo -n > _檔案.ext_` 指令有相同的效果，而檔案的三項時間戳記，則會一致設為目前的時間。

==== 檔案名稱
Linux 的檔案名稱長度一般不能超過 255 個字元。由於一個常用漢字在 UTF-8 編碼下，相當於以 3 個字元來表示，
因此檔案名稱長度的限制，會小於或等於 85 個漢字。

雖然依照規範衹有斜線(`/`) 與空字元(`\0`) 在檔案名稱中禁止出現，但是實務上都會儘量避免使用〈命令列模式：命令殼〉中提到的後設字元。
檔名後段常會以句號(`.`)分隔出一組較短的字串，習慣上稱之為**副檔名**(file extension)。
它的功用主要還是在方便使用者分辨檔案類型，對於作業系統本身並沒有意義。
大部分的軟體衹會將它做為參考，並不會因為副檔名而決定了處理的方式。

習慣上以句點做為起始的名稱，會被多數的應用軟體視為預設隱藏，而需要額外的選項或設定才會顯示。
這項性質並非來自系統層級，而是命名規則特例所產生的約定俗成效果。
其中還包括兩個特別的目錄名稱：.（單一句點）代表當前的工作目錄；..（雙重句點）代表上一層目錄。
除了單獨做為命令的參數使用，也可以混雜在路徑名稱之中。

=== 常用命令
目錄其實可以視為一種特殊的檔案，因此許多命令都能混合接受目錄與檔案做為參數。
不過仍然有些命令衹能用來操作目錄，而有些命令衹能用來操作檔案。

==== 檔案系統操作
在之前的小節中提到，檔案系統階層是依靠掛載來加入新的檔案系統裝置。
既有檔案系統階層中的任何一個空目錄，都可以做為**掛載點**(mount point) 使用。就好像在一段光
秃秃的枝條上，再嫁接上一束外部的枝葉。
掛載與取消掛載的外部命令分別是 `mount` 與 `umount` ：
====
root# *mkdir -p _/opt/DATA_* +
root# *mount _/dev/sdb1 /opt/DATA_* +
root# *cp -a _/root/DATA/++*++ /opt/DATA_* +
root# *umount /opt/DATA*
====
WARNING: `umount` 命令的拼法需要特別注意。

CAUTION: 掛載到非空的目錄並不會產生錯誤，但會使目錄中原本的內容被隱藏直到卸載。

有些輸出中含有（檔案、目錄、分割區）容量大小的命令，如果加上了 `-h` 選項，
代表的並不是常見的**求助**(help)，而是**易讀**(human-readable) 的意思。
這會將輸出中與位元組數量有關的數值，以 1024 的單位倍數呈現。

如果想要知道檔案系統的大致情況，最常用的是外部命令 `df`(disk freespace)。
====
user$ *df -h* +
檔案系統　　　　 容量　　已用　 可用　已用% 掛載點 +
devtmpfs　　　 4.0M　　　0　4.0M　　 0% /dev +
tmpfs　　　　　　16G　　　0　 16G　　 0% /dev/shm +
tmpfs　　　　　 6.3G　1.5M　6.3G　　 1% /run +

以上例而言，其中的 4.0M 代表的是 4.0MiB 亦即 4.0x1024x1024 位元組，而非 4.0x10^6^ 位元組。 

至於個別目錄分支所佔用的儲存容量，可以利用外部命令 `du`(disk usage)，搭配適當的引數獲得：
====
user$ *du -hs $HOME/+*+/* +
166M　　/home/user/bin/ +
4.0K　　/home/user/Desktop/ +
1.5G　　/home/user/Documents/ +
925M　　/home/user/Downloads/
====

==== 目錄操作
列出目錄內容的外部命令 `ls`(list) 是最常用的命令殼動作。
對於想要列出內容的路徑，就將一至多個名稱接在後方做為參數。
如果不加任何參數，則會列出工作目錄的內容。

基本的 `ls` 命令會以多欄對齊的方式，將內容顯示於終端機畫面上。
如果加上最常用的 `-l` 選項，則會改為每項檔案或目錄都佔一列的詳細顯示方式。
為了方便使用者輸入，多數的 Linux 散佈版會將 `ll` 設定成 `ls -l` 的別名。
另外還有取消預設隱藏的 `-a` 選項，能夠將以 `.` 開頭的檔案或目錄也顯示出來。

內建命令 `cd`(change directory) 用來變更工作目錄的指標，也會相應改變 `PWD` 變數。
如果提示字串中含有工作目錄的話，就會更新為目前的值。
由於 `cd` 與當前的命令殼行程緊密地連結，因此做為相容目的而存在的 `/usr/bin/cd` 外部命令，甚至連替補功能的角色都不容易扮演。

`cd` 後方通常接著一個目錄的路徑，可以是絕對路徑或相對路徑。
使用者當然也可以分層分批地多次輸入命令，最後仍然會到達目標的目錄位置。
一般使用 `cd` 的原因，是希望在之後輸入其他命令的參數時，可以使用相對路徑的簡單名稱；
或是在執行某些外部命令時，方便它在工作目錄中搜尋並調用相關的檔案。

實務上常常使用 `cd ..` 回到上一層目錄，或是執行 `cd ../_目錄名稱_` 移動到同層的其他目錄。
當 `cd` 單獨使用一個短橫線(`-`) 做為參數時，特別表示回到之前的工作目錄。
如果不加任何參數，則會依照變數 `HOME` 的值，將工作目錄變更回使用者的家目錄。

目錄的建立與刪除分別需要 `mkdir` 與 `rmdir` 內建命令。
如果 `mkdir`(make directory) 使用單一名稱做為參數，表示在工作目錄中以該名稱建立一個空的子目錄；
而以目錄路徑做為參數時，可能需要加上 `-p` 選項，才能依次建立各層的目錄，避免中間目錄不存在的錯誤發生。
需要注意的是， `rmdir`(remove directory) 的目標目錄，必須是不含檔案或子目錄的空目錄。

==== 檔案操作
有許多的方法可以用來新增一個檔案：
如果想將檔案拷貝一份，需要執行像是 `cp _檔案.ext_ _檔案.new_` 的指令；
但是如果想變更檔案名稱或是覆寫檔案，就要輸入類似 `mv _檔案.ext_ _檔案.new_` 這樣的指令。
指令 `ln _檔案.ext_ _檔案.lnk_` 會從 `檔案.ext` 建立出一個名為 `檔案.lnk` 的硬式鏈結檔案。
而指令 `ln -s _檔案.ext_ _檔案.lnk_` 則會建立出一個符號鏈結檔案。

想要刪除檔案時則需要使用 `rm`(remove) 命令，後面接著一至多個檔案名稱。
`rm -r` 指令可以接受目錄做為參數，能夠刪除即便含有檔案或子目錄的非空目錄。
利用指令 `mv _檔案.ext_ /dev/null` 將檔案搬移到假擬的空裝置，也同樣具有刪除的效果。
以上的方法都衹會斷開檔案內容區塊在目錄中的連繫，並不會抹除檔案在儲存媒體上的實際內容。

搜尋檔案或目錄主要有兩種方法：使用 `locate` 或是 `find` 外部命令。
執行 `locate` 命令前，需要先以超級使用者身分執行過 `updatedb` ，這會建立一個檔案系統階層的名稱資料庫，然後就可以利用 `locate` 加上部分名稱字串做為參數進行搜尋。
`find` 命令則是採取比較直接的方式，從使用者指定的目錄開始，比對以下層級的所有檔案或目錄，並依據指令中的條件過濾出要顯示的結果。
====
user$ *sudo updatedb* +
user$ *locate _ezusb.so_* +
/usr/lib64/pcsc/drivers/ezusb.bundle/Contents/Linux/ezusb.so +
user$ *find / -name _ezusb.so_* +
/usr/lib64/pcsc/drivers/ezusb.bundle/Contents/Linux/ezusb.so
====

當系統的儲存空間用盡，可能會出現很多奇怪的狀況：無法開機、無法顯示、無法登入、無法執行程式…等等。
最好是利用外部系統開機，再掛載目標裝置後進行清理，例如藉由執行 `find _/mnt_ -size +1G` 指令來尋找巨大檔案。
