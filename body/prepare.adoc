== 準備工作
測試之前除了顯而易見的硬體準備之外，軟體基礎設施的建立同樣相當重要。
人員本身關於軟體的概念以及對於工作的認知，其實也都會在無形之中影響著測試的進行。
本書內容並未涉及測試計畫的制訂，期望讀者在熟悉測試項目之後，能夠進一步邁向整合的目標。

=== 硬體組裝
組裝、測試場所的環境與工具，對於測試的進行也非常重要。
待測機器應放置於可移動的層架，各層以略大於待測機器的空間分隔，也要考慮到插拔線路與周邊替換等操作的方便性。
大型測試用治具應放置於不可移動的較大分層架中，通常位於桌面中央以適於共用。
測試桌面必須是不良導體、不易滑動、不易燃的材質，尤其注意不可有金屬的釘孔或邊條，還要避免容易產生靜電、磁場的環境。

==== 電源開關
當主機板上採用的不是焊接好的開關型式，就需要搭配符合接頭的開關元件，一般來說都有防呆機制而問題不大。
如果臨時找不到，也應避免使用金屬將開關插座中的針腳短路的方法。
萬一接通了錯誤的正負極腳位，有可能會為了一時的方便而損壞主機板。

==== 主機板電池
有時故意將電池拆下，等待數分鐘或是將電池座中的正負接頭短路，
可以重置系統韌體因誤設而造成的混亂內容。但是如果在未留意的狀況下，
電池的電力耗盡或是接點接觸不良，反而會導致系統韌體設定的不正常行為，
造成後續軟體出現無法預期的錯誤。

==== 中央處理器
精簡型電腦系統中的處理器，大多採用**單晶片系統**(SoC) 的型式，於工廠生產時便已焊接於主機板之上。
除了少數系統使用**處理器模組**(SoM) 的外插型式，測試人員通常並不需要自行安裝。

如果需要對於試作生產的裸板進行測試時，最常遇到散熱相關的問題。
要注意散熱膏與散熱鰭片的安裝是否正確，否則處理器過熱會導致測試中出現許多不穩定的狀況，
必要時應該準備外加的風扇或冷卻噴劑。

==== 記憶體
雖然記憶體的世代變換非常快速，自 DDR2...DDR5 一路以來不斷更新，
但是不同規格的模組都有不同的防呆缺腳，因此安裝方面相對簡單。
有時需要注意母板與模組之間，彼此的低電壓規格是否搭配。另外，測試用的記憶體模組，容易因為頻繁的插拔而
沾染到一些油漬或灰塵，可以利用橡皮擦來清潔金手指的部分。

==== 固態硬碟
固態硬碟從外形上主要分為晶片內包的封裝型式，以及晶片外露的模組型式。
封裝型式大多採用 SATA 或 USB 介面，前者應注意是否需要外加一條電源線；
模組型式的介面則可能是 SATA Slim、mSATA 或 M.2 的插口。
對於測試用的治具，除了注意金手指部分的清潔之外，還要避免過高的磁場與過於頻繁的讀寫，
才能有效延長儲存裝置的使用壽命。

==== 周邊裝置
外接裝置的接頭有方向之分，通常標誌凸起的圖案會朝上方、離主機板較遠的一側，或是配合大拇指施力時按住的地方。


=== 軟體概念
在介紹一些重要的軟體概念之前，我們要先對軟體的定義有所認識，
而藉由它的相對概念－硬體來理解將會容易得多。
硬體一般而言是指那些看得到、摸得到的實體，
實際上來說就是由零件（機械構造、電子元件）所組成的功能集合。
軟體相對於硬體則是由符號（程式、資料）所組成的功能集合，
物理上是以電磁（或光碟上的凹點）形式存在於媒體上的數位記錄。
而操作電腦就是藉助硬體動作，執行程式去處理資料的過程。

==== 系統階層
如果我們將軟體與硬體簡化為彼此堆疊的方塊，合起來便構成一個完整的電腦系統。
從底部一直到頂端接觸使用者的部分，就是由硬體加上軟體所形成的系統階層。

image::sys-hier.png[width=60%,align="center"]

緊鄰硬體上方有一塊特別的部分，這塊具有雙重特性的地方稱之為系統韌體（舊式的 BIOS 或新式的 UEFI）。
雖然它也是一組程式，但是被刷錄在晶片中後，就幾乎不會再去更動。
內核軟體之中還有一群也被稱作韌體的檔案，主要記載了特定裝置的可變參數資料，提供內核的裝置驅動程式調用。對於嵌入式系統而言，由於它的軟體部分
通常容量較小、通用性較低、變動機會較少，因此常常將軟體視為硬體的附屬，所以也會將軟體部分稱為整體系統的韌體。

軟體部分可以分為內核、系統函式庫、應用函式庫、命令殼與應用軟體，彼此之間都存在著連通介面，而狹義的**作業系統**(operating system) 即包括了應用軟體以外的軟體部分。
軟體底端階層的內核，是整個軟體系統最基礎的部分。
由於目前內核的架構採取與驅動程式緊密結合的方法，雖然可以確保每次內核釋出時原有驅動程式的相容性，卻也造成新增或修改驅動程式時的不方便。
系統函式庫包括了 C 語言函式庫、數學函式庫，以及其他支持系統基本運作的重要函式庫。
它們承接了應用函式庫的呼叫，再將其轉化為對於內核功能的呼叫。
應用函式庫與系統函式庫的區別並不十分明確，應用函式庫通常不是系統運作所必需，而是做為支持特定類別應用程式而存在。
應用軟體是負責軟體與使用者實際接觸的部分，命令殼與應用軟體中的桌面環境則是做為與使用者之間的介面：
不但將底層軟體的結果輸出給使用者，同時將使用者的輸入回送給底層的軟體去處理。

==== 框架
**框架**(framework) 是指編寫程式時所遵循的特定規範或規則，由一組互相搭配的系統函式庫與應用函式庫組成。
通常框架於開發初期，都有其偏好適用的程式語言（例如： MFC&C, Qt&{cpp}, Glib/GTK&C），
甚至是當創造原生(native) 語言時，做為其標準的開發框架而被同時提出（例如：.NET&C#, Cocoa&Objective-C）。
所謂的泛用框架，代表它也可以支援原生語言之外的程式語言。
也就是說，還是有機會利用自己熟悉的語言來撰寫符合泛用框架的程式，但是功能性與便利性可能就比不上原生語言。

依照框架來開發軟體有許多優點，例如：有固定的執行邏輯可以遵循、有標準的程式結構可以套用、有完整的函數功能可以呼叫。
雖然限制了一些可能性，但是能夠大幅增進開發的效率。
假如新的平臺上也有這個框架的實作，那麼原始碼最多衹需要小幅修改，就能在新平臺上編譯運行。
除非是像 Spring&Java, .NET&C#, Flash&ActionScript 之類達到了虛擬平臺層級的框架，
才能夠滿足所謂跨平臺的要求。

==== 系統平臺
要說明平臺的概念，需要先參照一下之前軟體在上、硬體在下的系統階層圖，平臺就是由多塊階層組合而成。
系統平臺可以分為軟體平臺與硬體平臺：軟體平臺建立於硬體平臺之上，而應用軟體則建立於軟體平臺之上。

硬體平臺由一群特定硬體所構成，成員包括中央處理器、主機板、晶片組、前期載入程式與周邊裝置。
**中央處理器**(CPU) 是硬體平臺的重要代表元件，它的常見架構種類包括 X86, ARM, PowerPC, MIPS, RISC-V，
所處理的位元長度目前多為 32 或 64，這是由 8-bit, 16-bit 一路發展上來的，未來也可能會普及更長位元長度的架構。
各類軟體用來標示硬體平臺的代號也很繁雜，就以作業系統內核對其的稱呼為例：

[square]
* 三十二位元(32-bit)：i386, armhf, powerpc, mips, riscv
* 六十四位元(64-bit)：x86_64, aarch64, powerpc64, mips64, riscv64 +

其中 X86 架構（包括 i386, x86_64）屬於**複雜指令集電腦**(CISC)，其餘架構歸類為**精簡指令集電腦**(RISC)。
一家晶片商可能會生產多種不同架構的產品，而同一種架構的 CPU 也可能有多家廠商生產。
X86 架構所搭配的前期載入程式，一般燒錄於主板的晶片上，早期搭配 16-bit 的舊式 BIOS，目前的趨勢則是採用 64-bit 的新式 UEFI。
其他處理器架構的主板可能並未配置 BIOS 或 UEFI 晶片，因此需要類似 uboot/redboot 的程式做為前期載入程式。

軟體平臺可以簡單地理解為：具有相同應用程式介面的一群作業系統。
常見的作業系統分類之下又有許多小的作業系統分支，包括名稱不同但彼此相容的作業系統，
例如 MS-DOS/FreeDOS, Windows/ReactOS, Fedora/Ubuntu 各對可以視為相同的軟體平臺。
軟體平臺同樣會依其內部處理的位元長度，區分為 16-bit, 32-bit, 64-bit...等等類型。

一種軟體平臺配上一種硬體平臺就形成了一種系統平臺，所謂 lintel 指的就是 Linux 加上 X86 的組合。
軟體平臺與硬體平臺的搭配，主要須注意所處理的位元長度是否符合。
由於硬體設計時多半有考慮到向下相容，因此實務上軟體平臺處理的位元長度，至少必須要小於（且相容）或等於硬體平臺處理的位元長度。
Intel 的 IA-64 架構本來是做為 IA-32(i386/i686) 的 64-bit 版本，但是因為設計時未考慮向下相容，
反而隨著代號一齊沒落，將接班地位讓給了 AMD 主導的 AMD64(x86_64/X64/EM64T)。

平臺依存性通常指的是：二進位可執行檔案衹能在特定系統平臺上運行。
而**跨平臺**(cross-platform) 軟體則意指該軟體可以在不同平臺上運行，一般分為下列幾種情況：

[square]
* 跨平臺編譯語言原始碼：各平臺都有編譯器的實作（C, {cpp}）
* 跨平臺編譯語言執行檔：達到虛擬平臺程度的框架（Java, .NET）
* 跨平臺解譯語言指令稿：各平臺都有解譯器的實作（Python, Shell）
* 跨平臺網頁語言指令稿：以瀏覽器做為虛擬平臺（JavaScript）

==== 虛擬機器
**虛擬機器**(system virtual machine) 是指利用軟體所構建出來的硬體平臺**仿真器**(emulator)，
通常用來仿造既有的真實硬體型號，當然也可以做為設計完新硬體、未製造出實體之前的測試機器。
在虛擬機器上安裝軟體平臺之後，才能成為完整的系統平臺，可以執行適用於該平臺組合的程式。

從軟體開發角度而言，仿真器與**模擬器**(simulator) 的差別在於：
前者建構了硬體層面 （例如 QEMU 自由軟體），而後者衹在軟體層面實現。
另一種說法則是：仿真器較着重細節，藉此受測的對象會以為處於真實的環境；
模擬器則較着重概念，因而受測的對象能夠認知到這並非真實的環境，甚至其本身亦為簡化過的版本。


=== 伺服服務與程式
進行測試時所需的資料檔案，一般會放在內部網路的檔案伺服器上，可能是以符合 SMB 或是 FTP, HTTP, HTTPS 的協定形式分享出來。
如果要操作遠端系統，則可以選擇建置文字介面的 SSH 或是圖形介面的 RDP, VNC 服務。
與 PXE 功能相關的 DNS, TFTP, DHCP, NFS 等服務，將在〈開機前執行環境〉中說明，而 CUPS 服務則會在〈輸入輸出介面〉中介紹。

[.center,width="60%",cols="2,2,3",options="header"]
|====
|網路埠號|服務名稱|服務用途
|20, 21|FTP|檔案傳輸
|22|SSH|安全遠端登入
|53|DNS|網域名稱解析
|67, 68|DHCP|動態分發網路位址
|69|TFTP|簡易檔案傳輸
|80|HTTP|提供一般網頁
|443|HTTPS|提供安全網頁
|631|CUPS|印表機管理與連網
|2049|NFS|網路檔案系統
|3389|RDP|遠端桌面登入
|5900|VNC|遠端桌面存取
|====

有些時候網路服務無法使用，是由於本機時間與伺服器相差太大，
還應先確認來源機器的網路位址，並且與目標機器有網路相連（可用 `iputils` 套件中的 `ping` 命令測試）。
問題的原因也可能與以下設定相關：兩端語區設定、協定的主動╱被動模式、SELinux、防火牆、磁碟空間使用配額。

==== 檔案服務
由於 FTP 協定規格對於資訊安全的防護較弱，因此網路上有逐漸減少這類服務的趨勢。
常用的伺服軟體套件是 `vsftpd` ，提供同名的 Systemd 類型服務，預設的資料根目錄則位於 `/var/ftp/` 。
服務組態檔中比較重要的部分包括：

./etc/vsftpd/vsftpd.conf
....
...
anonymous_enable=YES
local_enable=YES
write_enable=YES
local_umask=022
...
....
其他可選的 FTP 伺服軟體，還包括 `proftpd` 與 `pure-ftpd` 。
搭配的客戶端程式則能選用文字介面的 `ftp`, `lftp`，或是圖形介面的 `gftp`, `filezilla` 。

標準的 HTTP/HTTPS 伺服器安裝，請參考下一章〈開機前執行環境〉中相關的說明。
如果衹是為了一般測試的目的，不妨考慮安裝 `lighttpd` 套件。
它的組態檔案位於 `/etc/lighttpd.conf` ，相較於標準網頁伺服器會簡單許多。
另外想要測試網站是否支援 TLS 1.2 的話，可以試試：
====
user$ *openssl s_client -connect __網站網路位址__:443 -tls1_2*
====

除了啟動正式的網頁伺服服務之外，也可以執行臨時的網頁伺服程式來提供檔案分享。
假設伺服器中想要分享的檔案位於當前的目錄，可以選擇安裝並執行 `mongoose` 命令，或是以 Python 語言提供的模組來執行 `python3 -m http.server 8000` 指令。
客戶端就能利用 `wget` 命令來下載個別檔案，或是取得臨時分享目錄中的所有檔案：
====
user$ *wget http://_網路位址_:8000/_檔案名稱_* +
user$ *wget -r -nH http://_網路位址_:8000/*
====

源自 Windows 作業系統的 SMB 服務，新版內核已經提供了一個 `ksmbd` 模組，可以部分取代傳統使用者空間中的 Samba 軟體。
不過，大多數的 Linux 系統衹需要扮演客戶端的角色，藉由 `cifs-utils` 套件中的 `mount.cifs` ，以及自動載入的 `cifs` 內核模組，就可以掛載並存取到 SMB 服務所分享的檔案：
====
user$ *mount.cifs //_網路位址_/_分享名稱_ _掛載點_ -o username=_使用者名稱_,password=_使用者密碼_*
====

==== 原始碼管理
*Git* 是一種**原始碼控管**(SCM) 軟體，提供分散式控制與管理原始碼版本的服務。
如果測試環境能連接網際網路，可以直接利用 Github.com 的服務，甚至考慮購買私有服務以增強保密安全。
若是想要自行建立伺服服務的網頁應用，則有 Gitlab, Gogs, Gitea 等開放原始碼軟體可供選擇。

在客戶端安裝完 `git` 套件之後，先進行以下設定：
====
user$ *git config --global user.name _使用者姓名_* +
user$ *git config --global user.email _使用者電子郵箱_*
====
一旦獲得了伺服端的登入賬號之後，請先建立一個空的專案，接著在客戶端變更工作目錄到想要放置各個專案的地方。
然後將 Git 伺服器上的專案**複殖**(clone) 到本地機器中，並進入新建立的專案目錄以開始開發作業：
====
user$ *mkdir _Git_* +
user$ *cd _Git_* +
user$ *git clone https://_伺服器位址_/_使用者賬號_/_專案名稱_.git* +
user$ *cd _專案名稱_*
====
在開發目錄中可以進行新增、編輯、刪除等檔案操作，或將已有檔案拷貝到這裡。
當修改到了一個功能階段時，應該將成果儘快提交回伺服器，而不需要執着於一定的時間間隔或修改數量。
在專案目錄中輸入：
====
user$ *git add .* +
user$ *git commit -m 'create files and fix bugs'* +
user$ *git push origin master*
====
然後依照指示輸入賬號與密碼，就會將本地原始碼庫中的變更整合、壓縮並上傳到伺服器。
下次要繼續開發之前，應該在開發目錄中先匯入管理伺服器上其他開發者的提交記錄，以保持與最新的版本同步：
====
user$ *git pull origin*
====

==== 遠端登入
`sshd` 是使用 SSH 協定的遠端登入服務，已經幾乎完全取代舊式的 Telnet 服務。
附屬於 SSH 協定的還有 SFTP 服務，預設伺服端的設定同樣是禁止以超級使用者身分登入。
無法登入時要先確定不是因為鍵盤接觸不良所導致的輸入亂碼，或是因為使用者登入後自動執行過程出現錯誤所造成。

傳統由 X 視窗顯示管理程式所提供的 *XDMCP* 連線方式，目前必須要修改組態檔才會啟用：

./etc/gdm/custom.conf
....
...
[xdmcp]
Enable=true
Port=177
...
....
並且可能要刪除檔案 `/usr/share/polkit-1/org.freedesktop.color.policy` ，以避免連線時要求建立色彩描述檔。
客戶端的連線指令範例如下：
====
user$ *Xephyr :1 -fullscreen -br -bs -render default -dpi 96 +xinerama -reset -terminate -query _伺服端網路位址_*
====
其中的 `Xephyr` 將會扮演一個巢狀 X 視窗顯示伺服程式的角色。

Linux 系統是以 `xrdp` 套件來提供使用 RDP 協定的遠端桌面，安裝後即會啟動 `xrdp` 服務。
連線此服務時會執行自帶的顯示管理程式，使用者登入後，接著執行系統預設的作業階段管理程式。
因此在伺服端必須注意，先將其鏈結至想要的實際作業階段管理程式：
====
root# *update-alternatives --set x-session-manager _/usr/bin/xfce4-session_*
====
上述範例即會由作業階段管理程式導引至 XFCE 桌面環境。
客戶端的 RDP 連線，則需要藉由 `freerdp` 套件所提供的 `xfreerdp` 命令。

==== 遠端桌面
Linux 還有一項**獨立計算環境簡單協定**(SPICE) 開放標準，用於虛擬環境的遠端桌面顯示，但較少見於一般應用。
以下僅列出設置伺服端時最重要的**網路橋接**(network bridge) 設定，提供使用者需要時參考：
====
root# *systemctl enable libvirtd* +
root# *systemctl disable firewalld* +
root# *nmcli connection add type bridge autoconnect yes con-name br0 ifname br0* +
root# *nmcli connection modify br0 ipv4.addresses _10.10.2.78/24_ ipv4.method manual* +
root# *nmcli connection modify br0 ipv4.gateway _10.10.2.254_* +
root# *nmcli connection modify br0 ipv4.dns _10.10.1.2_* +
root# *nmcli connection modify br0 ipv4.dns-search _yourdomain.com_* +
root# *nmcli connection del _eth0_* +
root# *nmcli connection add type bridge-slave autoconnect yes con-name _eth0_ ifname _eth0_ master br0*
====

如果衹有單純的提供遠端桌面，僅需安裝支援 VNC/RFB 協定的 `x11vnc` 伺服程式，然後輸入指令：
====
user$ *x11vnc -forever -repeat -passwd _連線密碼_ &*
====
即可透過其他系統的 VNC 客戶端軟體連至目前的桌面：
====
user$ *vncviewer -DotWhenNoCursor -passwd _密碼檔案_ _網路位址_*
====
上述的密碼檔案中已儲存了所需的連線密碼。

如果遠端桌面的性能要求不高，伺服端又不想安裝完整的伺服軟體，客戶端也希望衹需要網頁瀏覽器就能連線，那麼就可以考慮採用遠端桌面閘道服務：
像是開放原始碼的 DWService
footnote:[DWS 遠端桌面閘道服務 <https://www.dwservice.net/>]，
免費註冊時還可選擇「傳統字漢語」網頁介面，而於伺服端安裝自動執行的代理程式後即可分享控制：

|====

|image::dwservice-list.png[]
|====

或是嘗試自行架設 Apache Guacamole
footnote:[Apache Guacamole 遠端桌面閘道軟體 <https://www.dwservice.net/>]，
但也必須面對有可能被防火牆擋住，而無法提供外部連線的問題。

==== 防火牆
網路組態中的**防火牆**(firewall) 設置，主要在日常使用的開放環境系統中，負責阻斷未經允許的網路輸入輸出。
但是在封閉的測試環境下，有可能因為疏忽反而造成其他伺服程式的不便。
防火牆的軟體服務名稱為 `firewalld` 或 `ufw`，在 Systemd 系統初始模式下可以依照一般使用 `systemctl` 命令的方式，搭配 `stop` 或 `disable` 選項分別予以停止或停用。
即使確定之後不會用到防火牆功能，但由於套件依存關係而不適合直接移除，不過仍然可以利用 `systemctl` 命令的 `mask` 選項，來阻止其他軟體去啟動它。

==== 檔案傳送
偶爾需要拷貝別臺電腦中的檔案時，通常是以**隨身碟**(flash stick) 做為交換的媒介，
因為臨時要啟用一個檔案分享服務，或是先將檔案上傳到伺服器再下載，都似乎稍嫌麻煩。
如果需要互相傳輸檔案的機器都位於同一內部網路，其實還有更簡便的方法：
使用 `netcat` 套件中的命令 `nc` ，就可以實現互傳檔案的功能。
先在接收端輸入以下命令：
====
user$ *nc -l > _檔案名稱_*
====
此時接收端會進入等待狀態，直到發送端輸入命令：
====
user$ *nc _接收端網路位址_ < _檔案名稱_*
====
此時接收端就會開始接收傳輸並儲存成指定名稱的檔案。
對於需要指定網路埠號並且傳輸整個目錄的情況，接收端的命令可以改為：
====
user$ *nc -l -p 31337 | tar -xvf -*
====
相應的發送端指令範例則是：
====
user$ *tar -cvf - _目錄路徑_ | nc _網路位址_ 31337*
====
如果命令與輸入資料都正確，卻出現無法連線的錯誤訊息，有可能是被接收端的防火牆所阻擋。

有些散佈版還會提供工具軟體 `croc` ，可以更方便地在區域網路中傳檔，而且是由發送端先啟動。
輸入指令 `croc send --code _臨時密碼_ _檔案或目錄_` 之後，發送端就進入等待傳輸狀態，然後再由接收端輸入 `croc _臨時密碼_` 即會開始發送與接收。


=== 外部測試系統
除非測項的內容就是安裝作業系統到固定的儲存裝置，或是需要測量安裝後固定儲存裝置的開關機與存取速度，否則大部分的測試都能用外部測試系統來完成。
利用隨身碟或外接硬碟上的系統啟動有許多好處，可以在短時間之內，替換不同種類或散佈版的測試系統，而不需要每次都得重新安裝。

==== 自活系統
最簡單的方式就是利用各種散佈版所提供的自活系統，通常已經和散佈版的安裝系統整合在一起。
將下載來的 ISO 檔案燒錄到光碟上，或是以 `dd` 命令寫入隨身碟後，就能以其做為外部系統開機。
如果手上有多個 ISO 檔案，還可以先製作一支 Ventoy
footnote:[Ventoy: 新的可開機 USB 解決方案 <https://www.ventoy.net/>]
隨身碟，用來將所有的可開機 ISO 檔案收集、整合在一起。

如果想利用自活系統簡單、快速的特性，來執行特定的開機後單一動作，也可以嘗試進行自活系統的客製化。
以常見的自活系統 SystemRescue
footnote:[SystemRescue: 系統救援工具箱 <https://www.system-rescue.org/>]
為例：首先準備好 `systemrescue-12.01-amd64.iso` 並編輯所需的 `data/autorun` 與 `data/datafile` 檔案；
然後利用十六進位編輯程式（ `hexedit` 或 `ghex` ），將 ISO 檔案中的 `timeout=90` 字串改為 `timeout=00` ；
接著執行指令：
====
user$ *growisofs -Z /dev/sr0=systemrescue-12.01-amd64.iso* +
user$ *growisofs -M /dev/sr0 -J -R data/*
====
將 ISO 檔案以及附加替代檔案燒入，以**重新壓製**(remastering) 光碟，之後還能用 `dd` 命令將光碟內容備份成修改過的 ISO 檔案。

==== 實用系統
在測試前的準備階段，可以預先將測試用的各種作業系統，分別安裝於多支隨身碟上。
測試系統所用的使用者名稱與密碼，應該要與其他測試人員協調一致，以便於交叉複測的進行。

隨身碟最好能夠做成支援 BIOS/UEFI 雙重開機模式，其中的套件組成與操作環境，也可以根據需求與喜好進行調整：
例如，安裝原本並非預設的工具套件，甚至增加散佈版的第三方套件庫。
如果測項並未強制要求系統必須開啟 SELinux 功能，也可以將它設為停用，以減少影響其他功能的機會。有些散佈版安裝後預設停用「自動開啟網路」，
可以依據需要將它改為預設啟用。
另外，測試系統的運作模式可以直接設定成「效能優先」，並且關閉螢幕保護與螢幕鎖定，如此比較不會干擾測試的進行。

由於很多的測試指令都是在命令列視窗中輸入，因此必須要有一個舒適的操作環境。
首先要將終端機仿真器的圖示釘選於工作列或側邊欄上，才不會每次都要從應用軟體列表中挑選。
雖然以快捷鍵呼叫終端機仿真器也很方便，但是有可能無法儲存對於視窗大小、字型、顏色偏好的調整。
命令殼的提示字串以及命令別名的設定，將會影響操作時的流暢度，同時也要確認自己慣用的文字編輯程式是否已經安裝妥當。

==== 套裝軟體
本書所用到的測試軟體，主要都來自於散佈版本身的套件庫。
在後續章節介紹各類裝置時，將會依需要列出所使用的工具程式。

一般散佈版的官方套件庫中，可以找到 `phoronix-test-suite` 這套開放原始碼的自動測試框架。
它能夠自動進行多種軟硬體的質化與量化分析，是以 PHP 程式語言所編寫的效能評測軟體，並以網頁的形式輸出結果。
雖然得到的資料豐富，但是需要自行從中提取有用訊息，才能夠整合到個別的測試報告之中。

許多測試軟體公司也提供適用於 Linux x86_64 的套裝測試軟體，甚至有免費的試用版本。
例如 PassMark Software 出品的燒機測試軟體 BurnInTest
footnote:[BurnInTest 燒機測試軟體 <https://www.passmark.com/support/bitlinux_faq.php>] ，
與效能測試軟體 PerformanceTest
footnote:[PerformanceTest 效能測試軟體 <https://www.passmark.com/support/pt_linux_faq.php>] 。
下載並解開、安裝之後，執行時可能會出現找不到函式庫的訊息，需要依照指示先安裝相關套件。

其他像是 Greeks3D 釋出的 GPU 壓力測試軟體 FurMark 2
footnote:[FurMark GPU 壓力測試軟體 <https://www.geeks3d.com/furmark/>] 與較舊版本的 GpuTest
footnote:[GpuTest GPU 壓力測試軟體 <https://www.geeks3d.com/gputest/>] 。
還有由 Primate Labs Inc. 所製作的處理基準測試軟體 Geekbench
footnote:[Geekbench 處理基準測試軟體 <https://www.geekbench.com/download/linux/>] ，會將評測結果上傳發布於網頁，方便和不同平臺、不同組態的系統比較。
另外像是 UNIGINE 3D 繪圖引擎軟體開發商，也有提供多種不同的 3D 場景評測軟體
footnote:[UNIGINE 3D 場景評測軟體 <https://benchmark.unigine.com/>] ，讓使用者測試整機硬體在運轉壓力下的繪圖表現。


=== 個人準備
加強專業知識畢竟是老生常談，而提升英文能力則是另一個層次的目標，但是對於常常接觸到的外文命令與名詞，還是必須要有一定的了解。
知道縮寫名詞的原文，並且能夠正確地唸出來，不僅有助於自己的理解與記憶，還能確保與其他人溝通時沒有障礙。
相比於一般操作，測試人員面對命令列視窗的時間更多，因此具備英文盲打技能將會使操作過程更加順利。

==== 釐清需求
測試人員與提出需求的客戶之間，通常存在著多層的溝通中介。
轉發而來的客戶需求，有時會去除被認為無關緊要的地方，有時則會加上額外的解讀。
所以要設法弄清楚哪一部分是經過刪減、補充的資料，如果能夠讀到客戶需求的原文內容也會多點幫助。

測試內容的擬定應該是基於客戶的實際需求，把為了達到這些需求而必要的功能列舉出來，再將能夠確認這些功能的操作型手法做為測試的步驟。
如果評估計畫中的測項對於滿足需求沒有幫助，並且可能會耗費大量資源，測試人員應該要儘早反映；
如果認為計畫中缺少了達到需求的關鍵項目，在測試資源許可、不損害原本計畫的前提之下，可以考慮額外進行一下驗證。

測試結果在經過內部流程之後，常常到達客戶的手中衹有部分資訊，因此必須假設客戶可能不會得到完整的結果報告，所以說明文字中不應省略重要的已知情況。
如果覺得有必要讓客戶知道的部分，就需要適當地在報告中特別註明。

==== 測試要點
測試過程中不免會遇到難以解決的問題，通常代表著陷入了尋找對策→試驗對策→尋找對策→…的無限循環。
此時不妨參考下面所列的幾項要點，也許就能有所突破：

[square]
* 問題的發生不全然是技術因素：有可能是開發歷史的遺留，也有可能是人為的疏忽。
* 復現問題時的環境儘量維持原狀：不論是硬體或是軟體，都應該設法重現問題發生時的組態。
* 除錯流程需要確實記錄：不能因為除錯是在標準測試流程之外的步驟，就省略了規畫與記錄的動作。
* 不要執着認定問題的根源：電腦系統是由硬體、韌體、軟體與操作者共同構成的，問題發生的層級未必就是問題根源的層級。
* 相伴發生的事件未必是因果關係：看似關連的事件，也許根本不相關，或是其實具有共同的問題根源。
* 對策的強度可能會影響效果：不要太快放棄沒有作用的對策，再試試降低或增加強度後的效果。
* 造成問題的原因可能不衹一個：單一對策對於問題如果沒有幫助，也可能是因為它需要與其他對策共同施行才有作用。
* 窮盡法衹適合做為最後手段：儘量優先使用推論或抽驗的方式，而不要太早採取遍歷所有的組態或工具組合。
