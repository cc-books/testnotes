== 作業系統安裝
____
[.text-right]
飛行特技不一定比單車特技困難
____
所有在電腦硬體上執行的軟體，都可以包括在廣義的作業系統之中，而狹義的作業系統（或是稱為作業系統核心），則僅包含內核、基本系統函式庫以及基本應用程式。
各個階層之間依靠程式介面進行溝通，而與使用者互動的介面即為**使用者介面**(user interface)。

=== 系統比較
常聽到的作業系統包括：DOS, Windows, macOS, Linux, Android, BSD, …等等。
雖然較精確的說法是利用「使用者空間核心/內核空間核心」的組合來稱呼，但是習慣都以 Linux 來代表 GNU/Linux、以 Android 來代表 Android/Linux ...。
當我們想要比較作業系統之間的差異時，利用〈準備工作：軟體概念〉中的觀點可以得到幾個分類的依據：

[square]
* POSIX ↔ 非 POSIX：除了少部分的應用程式，其他都不同（明顯可見的差異之一為檔案系統）。
* **單內核**(monolithic kernel) ↔ **微內核**(microkernel)：內核與系統函式庫所包含的組成比例不同。
* GNU libc ↔ BSD libc：基本系統函式庫中的 C 標準函式庫來源不同。
* 文字模式 ↔ 圖形模式：應用函式庫與應用程式所包含的軟體不同，造成使用者介面的差異。

網路上有數不盡的作業系統相關資源，例如 DistroWatch
footnote:[DistroWatch 作業系統動態網站 <https://distrowatch.com>]
就是一個追蹤數以百計自由/開放原始碼作業系統動態的網站，
除了提供個別系統的鏈結與下載資訊，還有主要組成軟體的版本變動記錄，以及獲得關注的點擊率排名。

==== 散佈版
Linux 散佈版之間的差異，在於由不同的社群組織或商業公司，將各種函式庫與應用程式整合到一起。
除了都是基於 *POSIX*(可移植作業系統介面) 的框架標準之外，造成不同的原因來自專案主導者的政策，
影響了所包括軟體套件的廣度與深度，以及統合基礎套件後所得到的整體觀感。

一般來說，社群組織主導的散佈版收錄較多通用的軟體，提供免費或僅需材料費的安裝與使用（像是 Fedora 或 Debian）。
由商業公司所主導的免費散佈版（像是 openSUSE 或 Ubuntu），能夠找到比較豐富的文件。
如果想要專業的軟體以符合特定的需求，或是注重後續的維護服務，就須選擇商業公司所提供的付費版本（像是 RHEL, SLE 或 Ubuntu Pro）。
也就是說，各類的 Linux 散佈版之間，並沒有實質上的差異，僅是集成過程導致了適用環境不同而已。

==== 版本選擇
實務上而言，測試計畫無法包含所有的散佈版，因此必須做出取捨。
首先要考慮客戶的需求，如果有明確指定、或是客戶預計使用的系統是基於某個散佈版，那麼當然要將它列入測試名單之中。
對於通用型機種，則要考慮散佈版的普及度、易用性、更新頻率、社群支援以及符合標準等因素。

散佈版的更新頻率取決於負責的開發組織或個人，可能是半年一次、一年一次或是不定期釋出新版。
版本之間通常以版本編號來區分，或是搭配釋出代號以方便推廣或記憶。
編號多半由年月分或是單純的流水號所組成，而主要版本之間的小改版，則以附加一位小數做為區別。
一般的狀況下，測試是針對最近期的主要版本，並且更新至最新的小改版。
安裝完的系統可以在 `/etc/*-release` 各個檔案中，看到關於作業系統版本的基本資訊。

有些散佈版會標記所謂的**長期支援版本**(LTS)，表示該版本的維護時間，會比其他普通版本更長 (通常是指提供重要的安全性更新)。
因此限制以長期支援版本做為測試標準，也是一種較為穩定的選擇。

==== 硬體架構
這裡所謂的**架構**(architecture)，是指依不同晶片指令集而分類的一組中央處理器家族。
適用於某類硬體架構的軟體，也就同樣具備了是否支援某類架構的屬性。
理想情況下，程式的原始碼對於硬體架構具有通透性（除了組合語言之外），衹有在經過編譯之後才會綁定硬體架構。

早期 PC 使用的是 32 位元的 X86 架構中央處理器，包括 i386、i486、i586、i686 等等。
由於幾乎已不再生產此類硬體，因此許多散佈版都停止支援這些架構，在網路上也不容易找到舊版的安裝軟體。
如果真的需要安裝 32 位元作業系統，系統韌體也必須支援並且調整為舊式的 BIOS 開機模式。

==== 虛擬機器
安裝到虛擬機器時，不要選擇**無人值守**(unattended) 模式，否則難以對預設的操作程序除錯。
特殊組合按鍵可能需要調整成全螢幕模式才能接收，或是衹能透過虛擬機器軟體選單來間接發送。
安裝完成之後，經由套件管理系統或是虛擬機器軟體選單，進行**客端擴充**(guest addition) 套件的安裝。
虛擬機器的網路拓樸模式設定，也會影響與其他實體或虛擬機器的連線架構。

=== 安裝準備
除了測試目標機臺的完整組件之外（尤其是儲存裝置），需要的周邊裝置也必須在安裝作業系統之前準備好。
如果測試環境中並未架設〈開機前執行環境〉所介紹的 PXE 功能相關服務，就需要準備符合需求的安裝媒體。

==== 安裝媒體
ISO 檔案是包含光碟完整內容的映像檔格式，依照國際標準「ISO 9660」或稱之為**光碟檔案系統**(CDFS) 所構成。
早期的電腦書籍或雜誌中，常常會附贈實體的安裝光碟片。
但是目前的 Linux 散佈版，衹會在網路上提供 ISO 格式的映像檔 `*.iso` ，讓使用者自行下載後，再以燒錄機製作出安裝光碟。

安裝光碟的用法是將它放入光碟機後，再以光碟機做為開機裝置來進行安裝程序，因此必須將其設定成優先啟動裝置。
系統韌體中的裝置選項可能會出現 CD/DVD 以及 UEFI:CD/DVD，表示這片安裝光碟同時支援 BIOS 與 UEFI 開機模式。
因此實際啟動的裝置，還會受到開機模式的優先順序影響。

如果製作出來的安裝光碟無法讀取，除了確認映像檔與光碟的總和檢查值之外，
還需要檢查看看光碟的標貼(CDLABEL) ，是否與光碟中開機載入程式組態檔的內容相同。
如果安裝媒體檢查沒有問題、換一片光碟問題還是同樣發生，而且移到其他電腦卻正常運作的話，
可以考慮將光碟機接在不同的埠上（例如從 USB2 換到 USB3），甚至找一臺內接式的 SATA 光碟機試試。

「ISO 9660」規格並未使用開頭的 32K 位元組，因此可以填入額外資訊來模擬其他的儲存裝置。
由於隨身碟容量較大、價格便宜，又有重覆寫入的方便性，因此大部分的 ISO 映像檔製作時，都加上了支援 USB 開機的功能。
也就是說，使用者可以像處理磁碟映像檔(`*.img`/`*.dd`) 一樣，將 ISO 檔案以 `dd` 命令直接覆寫隨身碟，製作出可開機的安裝媒體：
====
root# *dd if=_作業系統安裝_.iso of=/dev/_sdb_ bs=4M*
====
但是如此也增加了媒體損壞而導致過程失敗的風險，因此建議同時要考慮安裝方式的備案。
例如使用另外的儲存裝置做為安裝媒體，或是使用精簡型的光碟映像檔（需要連上網路才能進行套件安裝）。
另外還有將 ISO 檔案內容以 `7z x _作業系統安裝_.iso` 指令解開後，利用 PXE 伺服器進行的網路開機安裝模式，以及本章稍後會介紹的還原備份映像檔方法。

==== 周邊裝置
對一臺新的硬體型號進行安裝作業系統測試時，周邊裝置的組態應該要在合乎需求的前提下儘量簡化，才能減少測試時的變因。
出現問題時可能需要特別留意的地方：

[square]
* 顯示裝置如果規格太新，可以換一臺比較普通的機型。
* 如果還有支援，試試其他類型的顯示輸出。
* 儘量不要透過轉接線來顯示輸出。
* 更換不同規格的 USB 連接埠(2.0/3.0)
* 更換不同位置的 USB 連接埠(前端/後端)

==== 系統韌體設定
安裝前先套用系統韌體的預設值通常是不錯的選擇，萬一安裝過程不順利或是安裝結果不滿意，才會有一個變更的基礎。
其中可能需要變更的設定包括：

[square]
* 開機模式(BIOS/UEFI)
* 安全啟動(SecureBoot)
* 快速啟動(FastBoot)
* 快速儲存技術(RST)
* SATA 模式(IDE/AHCI/RAID)
* USB 支援規格(1/2/3)
* eMMC 讀寫速度(HS200/HS400)

==== 儲存裝置
安裝前必須先確認儲存裝置的容量，這會決定性地影響安裝過程。隨著功能與複雜度的不斷提升，Linux 系統的大小也持續地增加。
8GB 的磁碟空間已經不敷常見散佈版的標準安裝所需（雖然安裝後實際佔用空間未必大於 8GB），或是考慮下節〈安裝過程：目標裝置〉中所提到的變通方法。
如果安裝軟體有提供精簡組態（例如：LXDE 桌面、基本 X 視窗、最小化系統）的選項，也可以採用安裝後再加入所需軟體的方式，來建構自己想要的操作環境。

測試系統所用的儲存裝置，當然是以全新品最為適合。
不過由於常常需要重複測試，仍是不可避免會用到曾經安裝過作業系統的裝置。
用過的儲存裝置如果殘留之前的設定（尤其是沒有經過初始化分割區表、整支用來儲存資料的隨身碟，或是曾經被 `dd` 命令直接寫入過光碟映像檔的磁碟），再使用時就可能會造成安裝軟體的誤判。
建議先以外部系統開機（使用自活光碟或是系統隨身碟），然後利用 `gdisk` 命令先將目標儲存裝置重建一份空的分割區表。
如果儲存裝置曾經是 LVM(邏輯卷冊管理器) 的成員，但需要保留舊資料而無法進行格式化，則可以試試 `lvremove`, `vgremove`, `pvremove` 等命令。

有些內建混合型磁碟的機型，會發生安裝軟體無法識別的情形，可能是因為其中的多顆磁碟曾經被加入 RAID 群組。
萬一也因為要保留舊資料而無法進行格式化，則需以外部系統進入後執行：
====
root# *dmsetup remove_all* +
root# *dmraid -x /dev/sdb* +
root# *dmraid -r -E /dev/sdb*
====
或是手動抹去最前端與最末端的 1024 位元組（2 個磁區）：
====
root# *dd if=/dev/zero of=/dev/sdb bs=1k count=1* +
root# *dd if=/dev/zero of=dev/sdb bs=1k seek=$+((磁區數-2))+*
====

更複雜的情況是想要保留儲存裝置上原有的系統，也就是安裝成多重開機的模式。
除了要確認可用空間足夠之外，開機載入軟體的正確安裝與否，將會是多重系統成敗的關鍵。
安裝順序較後的 Linux 系統，通常都能自動整合較先安裝的其他 Linux 系統，對於 Windows 系統也大多能正確識別，
但是對於 Android, BSD 等系統就需要特別注意。如果還想要使用原先安裝的開機載入軟體，
那麼先要確認它對檔案系統格式與大小的支援限制，以及安裝時必須小心不要將它覆蓋，之後恐怕也無法避免手動的調整。

=== 安裝過程
準備工作完成之後，就可以開始安裝作業，並再確認一下電力來源是否充足、網路連線是否穩定。

==== 安裝介面
使用者在安裝過程中面對的操作環境，通常是圖形、文字、網頁三種介面之一：

圖形介面::
標準的圖形安裝介面，通常是功能最完整、操作最方便的模式。
如果安裝媒體中附帶的驅動程式支援不佳，可以嘗試在開機時加入內核參數 `nomodeset` 來進入基本圖形模式。
有些安裝軟體還提供遠端桌面（例如：VNC）的功能，可以從其他電腦監控安裝過程。
文字介面::
由於顯示器硬體或驅動程式的限制，萬一實在無法進入圖形介面時，需要以文字介面做為控制操作的權宜之計。
有些安裝軟體在此種介面下，還另外提供了專家模式，允許在安裝過程中進行更細微的調整。
網頁介面::
有些安裝媒體啟動後不提供任何文字或圖形介面，而是直接建立一個網頁伺服器。
使用者需要透過另一臺電腦系統中的瀏覽器，造訪安裝目標所在網路位址的網頁，然後依照其中的步驟進行安裝。

==== 內核參數
依據安裝軟體所使用的不同開機載入程式，使用者在安裝媒體啟動時立即按下 kbd:[Tab] 或 kbd:[E]，
就有機會手動輸入各種不同的內核參數（或稱密技碼），用來調整內核的執行，以解決安裝過程中遇到的問題：

[square]
* `nomodeset`: 如果無法進入圖形模式（舊版使用 `xdriver=vesa` ）。
* `nortc`: 如果出現無法開啟 RTC 裝置的錯誤訊息。
* `pci=nomsi`: 如果裝置或內核較舊，導致系統喚醒有問題。
* `selinux=0`: 如果載入內核之後出現長時間中止。
* `extlinux`: 如果想要安裝另一種開機載入程式。
* `iommu=soft`: 如果載入內核之後鍵盤沒有反應。
* `video=VGA-1:d`: 如果系統錯將 VGA 當作主螢幕。

其他還有 `noapic`, `nolapic`, `acpi=off` 或 `acpi=force` 可以進行個別或組合嘗試。

==== 語言與地區
電腦系統中的**語區**(locale) 設定，包括「語言設定」與「地區設定」兩大類。
安裝時所選擇的語言，除了會影響安裝介面的顯示之外，也會成為安裝完成後系統的預設語言。
雖然系統語言可以隨時變更，但是由於安裝媒體的發行單位，未必充分驗證過英語以外的語言環境，
因此標準測試仍然應該選擇「en_US(英語－美國)」為優先的語區，除非客戶的確是在非英語環境中使用，
以避免複雜組態（例如：額外的字型、輸入法與空間需求）所造成的出錯風險。

鍵盤佈局則是以實際操作所用的型式為準，以免出現鍵面字元與送出字元不符的情況。
預設值會配合語言的選擇，如果操作起來不順利，就試試通用的「English(US)」佈局，
或是檢查一下 kbd:[CapsLock] / kbd:[NumLock] 的狀態是否正確。

**時區**(timezone) 也是安裝過程中與地區設定相關的項目，大多會預設電腦硬體時鐘使用**世界協調時間**(UTC)，
再依照時區設定與配合網路對時的服務，決定系統顯示的時間。
有些系統的安裝過程會提供使用者較多的選擇權力，但有些則會根據時區而自動決定其他相關的地區設定。

==== 目標裝置
如果硬體中包含了一個以上的儲存媒體（例如內建裝置加上外接裝置），尤其是為了後續測試的可攜性，
而決定將系統安裝在隨身碟之類的移動裝置時，必須注意安裝軟體是否預設選擇了想要的目標。
萬一已經完成了步驟，卻發現選擇的儲存裝置錯誤，通常可以藉由安裝軟體的上一步功能，回到之前的步驟來修正。
但是別忘了要同時變更開機載入程式的安裝位置，以免發生根檔案系統與開機載入程式不在同一個儲存裝置（除非是有意為之），
而造成無法單獨由目標儲存裝置開機的情況。

從安裝軟體自動偵測到的裝置名稱，可以初步確認是否指定了正確的目標裝置。
例如 `/dev/sda` 代表第一個偵測到的 SCSI 介面相容儲存裝置，包括傳統硬碟、固態磁碟、隨身碟，
而 `/dev/sda1` 則代表該項裝置上的第一個分割區。但是 eMMC 或是 SmartCard 類型的儲存裝置，卻有不一樣的裝置名稱。
例如 `/dev/mmcblk0` 是代表第一個此種類型的裝置，而 `/dev/mmcblk0p1` 則是其上的第一個分割區。

目標儲存裝置的剩餘空間不足，將會導致安裝軟體的中斷執行，若是剛好稍微低於最小需求，可以採取一些變通方法：
在軟體偵測剩餘空間之前，先插入額外的隨身碟或硬碟，等到偵測通過之後再拔除；
或者如果安裝媒體是先進入**自活系統**(live system)，再執行以指令稿語言編寫的安裝軟體，
那麼就有機會手動去修改一下程式中的變數值，以繞過空間檢測的限制。

==== 安裝類型
以目前將開機模式設定為 UEFI 的業界標準來說，儲存媒體的分割區模式需要採用 *GPT*(全域唯一識別碼分割區表)。
UEFI 相較於 BIOS 開機模式要複雜許多，後者一般搭配 *MBR*(主開機記錄) / *DOS*(磁碟作業系統) 分割區模式。
因此在安裝過程中，比較安全的做法是讓安裝軟體自動產生預設的分割區表（示例如下）：

[.center,width="80%",options="header"]
|====
|分割區	|掛載點		|分割區標籤	|檔案系統
|sda1	|/boot/efi	|ESP		|vfat
|sda2	|/boot		|Linux Native	|ext4
|sda3	|/		|Linux Native	|ext4
|sda4	|		|Linux Swap	|swap
|====

以上設定會將第一個 SCSI 相容儲存裝置全部分配給新的系統使用，原有的系統將被完全覆寫。
如果真的需要建立一個多重開機的系統，而安裝軟體又沒有提供保留舊系統的選項，
或是系統剩餘的儲存空間已經不夠讓新系統共存，那麼就衹好選擇手動的方式。

假如需要手動規畫分割區表，必須注意一定要存在一個 ESP 類型、並以 FAT 檔案系統規格化的分割區。
如果系統預期會頻繁開關機，而又非常重視妥善率的話，甚至可以考慮建立第二個 ESP 做為備援。
雖然預設會建立的第一個分割區，是大小約為 500MB 的 FAT32 檔案系統，不過如果 UEFI 與內核的相容性（容錯程度）較高時，
其實也允許分割區不在裝置前端、280 MB 或更小的容量、FAT12/FAT16 的檔案系統，甚至使用 MBR/DOS 分割區模式都是有可能被接受的。

其中的 `/boot` 掛載點還可以併入根檔案分支 `/` ，或是將 `/home` 從根檔案分支分離出來，掛載為獨立的分割區。
至於裝置末端的**置換分割區**也可以省略，尤其是當硬體系統具備充足的記憶體時，或是有些安裝軟體以建立**置換檔案**來代替。
其他的決定依據，可以進一步參考〈儲存裝置：檔案系統〉章節的說明。

==== 套件選擇
除了精簡型的網路安裝光碟之外，一般的安裝媒體都支援離線安裝，同時在偵測到網路連線時，可以選擇背景下載更新套件甚至自動更新。
某些安裝軟體還能讓安裝者根據自身需求，決定是否加入含有非自由軟體的套件庫。
如果安裝後重新開機發生問題，不妨試試改採離線安裝的方式，以方便確認問題的來源。

常見的散佈版都是基於套件的組合式系統，因此大多允許使用者在安裝時選擇套件。
安裝軟體通常會提供適用於「桌面」、「伺服器」、「特殊需求」或是「最小化」的整合類別，以方便使用者快速選擇，而不需要擔心套件之間的依存問題。
例如在「桌面」類別之下，包括了「GNOME」、「KDE」、「MATE」、「LXDE」…等選項；在「伺服器」類別之下，則包括了「網頁」、「資料庫」…等選項。
由於安裝光碟的製作趨勢是同時具備自活光碟的功能，因此在光碟容量的考量之下，仍有可能對各種安裝類別的選擇有所限制。

==== 命令列模式
某些額外的安裝需求衹能藉由輸入命令達到，此時就要設法先進入命令列模式。
對於先進入桌面環境再執行的安裝軟體，可以直接打開終端機仿真視窗，
否則就得試試按下組合鍵 kbd:[Ctrl]+kbd:[Alt]+kbd:[F2]，看看是否能切換到一個虛擬終端機。

額外需求像是重建**初始記憶體檔案系統**(initramfs) 檔案，用以強制加入驅動儲存裝置的內核模組：
====
root# *chroot /mnt/sysimage* +
root# *dracut -f --add-drivers sdhci-pci* +
====
上述指令會基於目前運行中的內核版本，並且在預設的路徑產生檔案。

如果這個方法有效的話，那麼在安裝成功進入系統之後，不要忘記修改 `/etc/dracut.conf.d/` 之中的組態檔，
才能在內核更新而 `dracut` 套件仍未修正時，讓所有之後產生的初始記憶體檔案系統都具備這些模組，以繼續保有儲存裝置的支援。

./etc/dracut.conf.d/local.conf
....
add_drivers+=" sdhci-pci "
....

==== 主機與使用者名稱
主機名稱要以滿足識別的需求為主，最好看見名稱就能夠知道是哪一臺機器。
如果安裝時並未連上網路，就要特別注意未來在區域網路中的名稱獨一性。

有些安裝程序允許衹設定超級使用者 root 的賬號（扮演系統管理者的角色），但是仍然建議設定一位一般使用者的賬號與密碼，
至少可以做為超級使用者賬號無法進行管理時的備援機制。
有些安裝程序則恰好相反，預設衹會建立一位一般使用者，但具備切換為超級使用者的能力。
同時，安裝程序還可能提供不需賬號密碼直接登入的選項，不過這項功能對於測試系統而言並不適合。

如果衹是做為內部測試用途，那麼建議設定賬號時使用 user 或 tester 之類容易記憶的名稱。
另外可以利用公司的名稱、電話號碼或是統一編號做為密碼，以方便其他人員後續測試。
如果是私人用途，習慣上會以英文姓名的簡化組合做為賬號，其中可以穿插數字或底線符號，但儘量不要以數字起始。
密碼應該包含多種類型的字元以及足夠的長度，並且注意要以自己容易記憶為前提。

=== 安裝後續
如果安裝過程順利完成，系統會要求使用者將安裝媒體退出後重新開機。
使用者必須確認於下次啟動時，新安裝的系統會是優先開機的分割區裝置。
萬一無法進入桌面環境，可能有以下幾種原因：

[square]
* 系統韌體設定的開機模式與安裝時不一致。
* 開機載入程式或系統根檔案系統的安裝位置不正確。
* 開機載入程式無法讀取此類型儲存裝置。
* 系統內核不支援或未載入儲存裝置的驅動模組。
* 系統內核對繪圖晶片的支援不佳。

請參考之前各節的說明與後續各章的內容，以尋求解決問題的方案。
安裝後的首次啟動，系統預設將會執行一些必要的設定。

==== 操作身分
雖然對於臨時的測試系統而言，直接以超級使用者賬號登入，在操作時會十分方便，卻無法準確模擬未來真實的使用情境。
安裝軟體可以設定，是否自動賦予一般使用者暫時切換為超級使用者的權力，亦即允許執行 `sudo`(superuser do) 命令，
並緊接著需要超級使用者身分的命令做為引數，然後回答該位一般使用者的密碼便可執行。
如果一般使用者不具備 `sudo` 的執行權力（未列於 `/etc/sudoers` ），則需要改為輸入命令 `su`(substitute user) 與建議的短橫選項(`-`) ，然後回答超級使用者密碼，才能夠完整切換成超級使用者身分。
例如：
====
user$ *sudo _dmidecode_* +
[sudo] password for user: +********+ +
... +
user$ *su -* +
Password: +********+ +
root#
====
如果具備 `sudo` 執行權力的一般使用者，想要連續以超級使用者身分輸入命令，但卻不知道超級使用者的密碼，
還可以結合上面的兩項命令，衹要回答一般使用者的密碼，就能夠維持切換成超級使用者身分：
====
user$ *sudo su -* +
[sudo] password for user: +********+ +
root#
====
有些散佈版在安裝完之後，預設拒絕直接以超級使用者身分登入。
部分系統衹要設定好超級使用者的密碼就會允許登入，其他則需要修改 *PAM*(隨插式認證模組) 的限制功能，相關的設定可見於 `/etc/pam.d/` 目錄之中。
另外某些系統服務或程式，也會被設定為必須要/不允許以超級使用者身分執行。

==== 時間調整
如果在安裝時沒有設定好系統時區或時間，可以進入系統後再執行變更的指令：
====
root# *timedatectl set-timezone _Asia/Taipei_* +
root# *timedatectl set-time _"2025-10-04 17:00:00"_*
====

如果更新或安裝作業不成功，請查看是否網路未通、防火牆造成阻擋，或是套件庫的設定不正確。

萬一安裝後的系統實際佔用空間，已經接近儲存媒體的大小，也就是說剩餘的可用空間很少，
那麼就要考慮清理不需要的套件，或是停用基於檔案的置換空間機制。
另外根據測試任務需求與個人操作習慣的不同，「省電設定」與「螢幕保護」也是安裝後通常需要調整的項目。

除了散佈版官方的套件庫之外，使用者還可以自行加入一些由組織或個人提供的非官方套件庫
footnote:[Fedora 社群最常用的 RPM Fusion 套件庫 <https://rpmfusion.org/>]
footnote:[筆者所建立的 RPM Sphere 套件庫 <https://rpmsphere.github.io/>]。
這些第三方套件庫包含了官方套件庫中未收錄的軟體，也許是使用授權不符合官方收錄的標準，亦或是單純因為沒有引起任何套件製作者的興趣。
從第三方套件庫網站取得安裝套件檔案後，以下列指令安裝（直接將檔案網址做為參數亦可）：
====
root# *rpm -Uvh rpmfusion-free-release-_43_.noarch.rpm* +
root# *rpm -Uvh rpmfusion-nonfree-release-_43_.noarch.rpm* +
root# *rpm -Uvh rpmsphere-release-_43_-1.noarch.rpm*
====

==== 備份與還原
所謂的**安裝系統**(installer)，本質上其實就是一套極簡的作業系統，衹是在系統初始化後，自動執行特定的安裝程式。
安裝過程結束後則會關閉系統，通常不會讓使用者有進到桌面或命令列模式的機會。
製作這類可以讓其他人使用的全自動安裝系統，需要藉由特殊的軟體，或是具備程式編寫能力與對於系統深入的了解。
如果衹是為了個人使用方便，則可參看〈儲存裝置〉中關於手動備份與還原的說明。

而介於兩者之間的半自動備份與還原動作，則有一些開放原始碼軟體可以達成。
例如：Clonezilla
footnote:[Clonezilla 磁碟複殖系統 <https://clonezilla.org/>]
是一款開放原始碼的磁碟複殖系統。
利用命令稿將許多磁碟映像相關的工具整合在一起（例如 `dd`, `parted`, `partclone`, `partimage` ），搭配基礎開機軟體而以自活光碟映像檔的形式釋出。

將映像檔燒錄到隨身碟或光碟之後，確認系統韌體中的設定，以其在來源裝置所在的機器上開機，而各項作業前，均要確認優先的開機裝置。
啟動系統後依據選單引導，進行磁碟的映像拷貝、備份、還原等作業，操作過程衹需要鍵盤而不需要滑鼠。
開機後建議插入額外的隨身碟，做為儲存映像檔目錄或是還原光碟映像檔的裝置。
還原時可直接以還原光碟映像檔所燒錄的自活裝置開機，或是以 Clonezilla 系統配合額外的映像檔目錄儲存裝置來完成。

==== 多重開機
將多個作業系統安裝於同一套硬體系統中，稱之為**多重開機**(multi-booting)。
開機模式雖然取決於系統韌體的支援程度，但是同一個儲存裝置上安裝的作業系統，仍然應該要使用同一種開機模式比較方便。

如果系統韌體無法提供適合的開機選單，就需要一個具備選單功能的開機載入程式。
這類軟體可以是安裝作業系統時所附帶，或是由使用者額外安裝。
因為各個開機載入程式的功能與相容性不同，使得安裝作業系統的先後順序也會影響多重開機的結果。


=== 整體效能
有些軟體提供了較為整體的測試流程，適合提供系統狀態的概觀給一般使用者。
整合測試通常會在剛安裝好的新鮮系統上進行，以避免系統使用過一段時間之後所累積的外部影響。
如果需要長時間運行程式，必須注意系統可用的儲存空間，是否能夠容納過程中產生的暫存檔案。

==== 組成資訊
整合性的系統硬體資訊顯示軟體中，運行於文字介面的包括 `lshw`, `inxi`, `hwinfo` ，都會印出大量的系統硬體資訊。
`hardinfo` 是 Linux 系統中相當重要的系統資訊與效能評測軟體，具備圖形化介面與豐富的內容，目前大多數的散佈版都收錄了第二代的 `hardinfo2` 套件：

image::hardinfo2.png[width=60%,align="center"]

==== 系統負載
內核專案中附帶提供的 `perf` 工具程式，可以用來啟用內核的效能監控。
套件 `sysstat` 包含了一些與系統效能相關的個別測試程式；套件 `glances` 則具備比較全面的效能監控功能；而套件 `sysbench` 則是比較側重系統在運行資料庫時的表現。
有些軟體具備測試效能的附加作用，例如執行 `openssl speed md5` 或 `7z b` 等指令後，其輸出也可以做為系統效能的參考指標。

圖形化介面的桌面軟體部分，除了舊式非常陽春的 `xload` 命令之外，在 GNOME 桌面環境中預設使用的是 `gnome-system-monitor` ：

image::gnome-system-monitor.png[width=60%,align="center"]

==== 壓力測試
由於壓力測試常常搭配溫升、電磁這類變動外部環境的測試，或是更嚴苛的項目（防水、防塵、防震），
因此並不會分開測試個別的元件，而是整體同時進行（包括中央處理器、記憶體、輸入輸出裝置）:
====
user$ *stress -c 4 -i 2 -m 2*
====
TIP: 另一個用法類似的新一代命令： `stress-ng` 。

或是利用系統中既有的簡單命令，讓它們去做一些耗費資源的無意義動作，藉此達到強制系統全速運轉的目的。
例如指令 `md5sum /dev/zero` 或 `dd if=/dev/zero of=/dev/null` 執行後，都會讓系統進入不休止的作業，並且持續施加壓力給中央處理器。

此時多半需要搭配系統監控軟體，以便即時顯示系統資源的使用情形。
在文字模式下可以執行傳統的全螢幕 `top` 命令：

image::top.png[width=80%,align="center"]

類似的命令還有 `atop`, `btop`, `htop` ，或是第三方 `bottom` 套件中的 `btm` 命令，均為受到 `top` 所啟發的較華麗版本。
